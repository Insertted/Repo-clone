# 🐳 Учебное пособие по Dockerfile: от основ до оптимизации

Это руководство поможет вам разобраться, как превратить ваш код в переносимый Docker-образ. Мы разберем синтаксис, лучшие практики и внутренние механизмы сборки.

---

## 1. Что такое Dockerfile?

**Dockerfile** — это текстовый файл, содержащий последовательность команд (инструкций), которые Docker использует для автоматической сборки образа. 

Представьте это как рецепт:
1. **Образ (Image)** — это готовое блюдо (неизменяемый шаблон).
2. **Контейнер (Container)** — это порция еды, поданная на стол (запущенный экземпляр образа).

---

## 2. Основные инструкции

### `FROM` — Фундамент
Каждый Dockerfile должен начинаться с `FROM`. Эта команда задает базовый образ, на котором вы строите свой.
```dockerfile
FROM node:18-alpine
```

### `WORKDIR` — Рабочая директория
Вместо использования `cd`, используйте `WORKDIR`. Все последующие команды (`RUN`, `COPY`, `CMD`) будут выполняться в этой папке.
```dockerfile
WORKDIR /app
```

### `COPY` и `ADD` — Перенос файлов
* `COPY`: Просто копирует файлы с вашего компьютера в образ. (Рекомендуется)
* `ADD`: Умеет распаковывать `.tar` архивы и скачивать файлы по URL.
```dockerfile
COPY . . 
```

### `RUN` — Сборка образа
Выполняет команды внутри образа на этапе сборки (установка зависимостей, компиляция). Каждая команда `RUN` создает новый слой.
```dockerfile
RUN npm install
```

### `ENV` и `EXPOSE` — Окружение и Порты
* `ENV`: Устанавливает переменные окружения.
* `EXPOSE`: Документирует, какой порт контейнер будет слушать (не открывает его автоматически на хосте!).
```dockerfile
ENV NODE_ENV=production
EXPOSE 3000
```

### `CMD` против `ENTRYPOINT` — Запуск контейнера
* `CMD`: Команда по умолчанию при запуске контейнера. Её легко перезаписать при старте.
* `ENTRYPOINT`: Основная команда, которую сложно перезаписать. Обычно используется для превращения контейнера в исполняемый файл.
```dockerfile
CMD ["node", "index.js"]
```

## 3. Жизненный цикл слоев и кэширование

Docker выполняет команды послойно. Если вы не меняли файл `package.json`, Docker пропустит шаг `npm install` и возьмет результат из кэша.

**Плохой пример:**
```dockerfile
COPY . .
RUN npm install  # Кэш сбросится при любом изменении любого файла кода
```

**Хороший пример (Оптимизированный):**
```dockerfile
COPY package.json package-lock.json ./
RUN npm install  # Кэш сохранится, пока не изменятся зависимости
COPY . .
```

## 4. Многоэтапная сборка (Multi-stage Builds)

Это лучший способ уменьшить размер итогового образа. Мы собираем приложение в одном образе (тяжелом), а запускаем в другом (легком).

```dockerfile
# Этап 1: Сборка
FROM node:18 AS build-stage
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Этап 2: Продакшн
FROM nginx:stable-alpine
COPY --from=build-stage /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

## 5. .dockerignore

Создайте файл `.dockerignore` в корне проекта, чтобы не копировать лишнее (логи, зависимости хоста, Git-папки). Это ускоряет сборку и уменьшает размер.

**Пример `.dockerignore`:**
```text
node_modules
npm-debug.log
Dockerfile
.git
.env
```

---

## 6. Шпаргалка по командам терминала

Для работы с вашим Dockerfile используйте следующие команды:

* **Сборка образа:**
    ```bash
    docker build -t my-app:v1 .
    ```
* **Запуск контейнера:**
    ```bash
    docker run -d -p 8080:3000 --name instance1 my-app:v1
    ```
* **Просмотр слоев образа:**
    ```bash
    docker history my-app:v1
    ```

## Лучшие практики (Best Practices)
1.  **Используйте теги:** Никогда не используйте `FROM node:latest` в продакшене. Используйте конкретные версии (например, `node:18.16.0-alpine`).
2.  **Минимизируйте количество слоев:** Объединяйте связанные команды `RUN`:
    ```dockerfile
    RUN apt-get update && apt-get install -y \
        git \
        curl \
        && rm -rf /var/lib/apt/lists/*
    ```
3.  **Запуск от не-root пользователя:** Для безопасности лучше создать пользователя и переключиться на него через `USER`.
